Compute Layer
AWS Fargate (ECS)
Crawling Engine runtime
Why needed: The Crawling Engine requires a persistent, long-running process — it holds a live Tor controller connection via stem, maintains SOCKS5 sockets, and rotates circuits. Lambda's 15-min limit and stateless cold-start model are incompatible with this. Fargate runs the crawling container with a Tor sidecar (separate container in the same task) exposed on 127.0.0.1:9050 / 9051, matching how CrawlingEngine.get_proxy_url() is wired. It scales the crawl fleet horizontally with ECS service autoscaling.

AWS Lambda
Short-lived agent steps
Why needed: The Content Analyst, Data Structurer, Tagging Engine, and Alert Generator are stateless, invocation-scoped — they receive S3 payloads, process them via Bedrock/OpenSearch, and return structured outputs. Lambda is ideal: pay-per-invocation, scales to zero, and integrates natively as Step Functions task states (replacing the current Pass placeholders in cdk_pipeline_stack.py). Each agent = one Lambda function with its own IAM execution role (already scaffolded in CrawlingEngineRole).

Agent runtime + managed KB
Why needed: AgentCore provides the declarative agent harness — memory, tool use, and reasoning loops — wrapping each agent class. Orchestration & Scheduling
AWS Step Functions (Express)
Pipeline orchestrator
Why needed: Already in cdk_pipeline_stack.py but using Pass placeholder states. In production, upgrade to Express Workflows (high-throughput, per-execution pricing) and replace Pass states with Task states invoking Lambda + ECS. Express handles the 5-minute cadence (>1M executions/month) without Standard Workflow costs. The state machine passes the S3 artifact key as input through each agent step, maintaining the pipeline's data contract.

Amazon EventBridge Scheduler
Crawl trigger (every 5 min)
Why needed: The existing events.Rule in CDK fires the Step Functions state machine every 5 minutes. In production, switch to EventBridge Scheduler (not just Rules) for timezone-aware scheduling, flexible invocation windows, and dead-letter queue support for missed invocations. Also use custom event buses to decouple crawl triggers from downstream processing triggers.

Alert delivery + backpressure
Why needed: Already in CDK (AlertQueue with DLQ). Networking & Security
Amazon VPC (Multi-AZ)
Tor proxy isolation
Why needed: Already in CDK with Public/Private/Isolated subnets. The Fargate crawling tasks run in Private subnets and reach .onion addresses through a NAT Gateway (Tor traffic exits to the internet via the NAT's EIP). Lambda agents in Isolated subnets reach AWS services via VPC Interface Endpoints (S3, DynamoDB, Bedrock, Secrets Manager, OpenSearch) — never traversing the internet. This is the security boundary that prevents dark web traffic from contaminating the analysis layer.

AWS IAM (per-agent roles)
Least-privilege execution
Why needed: CDK already creates CrawlingEngineRole with only S3 read/write + DynamoDB + Secrets access. Production should extend this pattern — one IAM role per agent with only the permissions that agent's code actually calls. E.g., the Content Analyst role only needs bedrock:InvokeModel and s3:GetObject. Use IAM Conditions to scope S3 access to the specific bucket prefix (crawl-artifacts/*) rather than the whole bucket.
All agents
AWS KMS (CMK)
Encryption at rest
Why needed: Dark web intelligence data is sensitive — stolen credential data, BIN ranges, SWIFT codes. The current CDK S3 bucket uses S3_MANAGED encryption. For a banking fraud use case, upgrade to Customer Managed Keys (CMK) — one CMK per data classification tier (raw artifacts, STIX objects, alerts). This gives audit trail via CloudTrail of every decryption event, and per-agent key policies prevent cross-contamination.

Amazon GuardDuty + Security Hub
Threat detection on the system itself
Why needed: A system that processes dark web content is itself a high-value attack target. GuardDuty monitors for anomalous API calls (e.g., unexpected Bedrock invocations, S3 exfiltration patterns), while Security Hub aggregates findings from GuardDuty, Inspector (container vulnerability scanning for the Fargate Tor image), and IAM Access Analyzer. This is the "security of the security system" layer — missing from the current CDK.

Pipeline health + agent telemetry

AWS X-Ray
Distributed tracing across pipeline steps
Why needed: The pipeline chains 5 agents across Lambda + ECS + Bedrock + OpenSearch. X-Ray traces the full execution path of a single crawl artifact through the pipeline — critical for diagnosing latency spikes (e.g., "Bedrock took 8s on this run") and identifying which agent failed in the Step Functions trace. Enable Active Tracing on Lambda functions and the Step Functions state machine; instrument Bedrock calls with the X-Ray SDK.

How the Services Connect Together

🕷 Crawl → Store path

EventBridge Scheduler fires every 5 min → triggers Step Functions Express
Step Functions → ECS RunTask → spins up Fargate task in Private subnet
Fargate app container uses Tor sidecar (127.0.0.1:9050) → crawls .onion via NAT Gateway egress
store_artifact() → S3 (raw content + annotation sidecar JSON)
Circuit breaker state written to DynamoDB AgentStateTable via VPC Gateway Endpoint (no internet)

🧠 Analyse → Structure path

Step Functions passes S3 key → Lambda Content Analyst invoked
Lambda reads artifact from S3 → calls Bedrock InvokeModel (Claude, via Interface Endpoint)
Bedrock Guardrails screens content before Claude processes it
Data Structurer Lambda → creates STIX bundle → calls Bedrock Titan for embeddings → writes to OpenSearch Serverless
STIX bundle also serialized to S3 for Bedrock Managed KB RAG retrieval

🏷 Tag → Alert path

Tagging Engine Lambda reads STIX bundle → applies ATT&CK + taxonomy → writes tagged bundle back to S3
MISP Integration Lambda calls Secrets Manager for API key → posts event to external MISP via VPC NAT
Alert Generator Lambda reads DynamoDB ConvergenceTable → checks TTP convergence window (24h TTL)
When 3+ items converge → publishes to SNS → fans out to SQS FIFO (SIEM), email subscription (SOC), and API Gateway webhook
CloudWatch + X-Ray traces entire execution; DLQ alarm fires to SNS (ops) on failure

🔐 Security planes

KMS CMK encrypts S3, DynamoDB, OpenSearch at rest — CloudTrail logs every decrypt call
IAM roles per agent — Crawling Engine can't call Bedrock; Analyst can't write to ConvergenceTable
VPC Interface Endpoints for Bedrock/Secrets Manager/OpenSearch — no data ever leaves the VPC for AWS API calls
GuardDuty monitors for anomalous behaviour (data exfil, unexpected cross-account calls)
Security Hub aggregates Inspector (container vulns), IAM Access Analyzer, and GuardDuty findings into one dashboard
