Dark Web Fraud Intelligence Agent — Production AWS Architecture

Multi-Agent Serverless Pipeline · VPC-Isolated · Encrypted at Rest (KMS CMK) · X-Ray Traced

How It Works — Architecture Walkthrough

A step-by-step guide to the pipeline, the AWS services that power it, and how they connect.

1

Scheduling & Orchestration — EventBridge Scheduler + Step Functions Express

Every 5 minutes, Amazon EventBridge Scheduler fires a trigger that kicks off the intelligence pipeline. Rather than running a single monolithic process, the work is broken into discrete, auditable stages managed by AWS Step Functions — specifically an Express Workflow, chosen because it handles high-frequency, short-duration executions at a fraction of the cost of Standard Workflows.

Step Functions passes a JSON payload — containing the S3 artifact key and execution ID — through each pipeline stage in sequence, handling retries, timeouts, and failure routing automatically. If a stage fails, the execution history is stored in CloudWatch Logs for debugging.

2

Dark Web Crawling — ECS Fargate + Tor Sidecar + NAT Gateway → S3

Step Functions launches a Fargate task inside a private VPC subnet. The task runs two containers side-by-side: the Crawling Engine (Python) and a Tor sidecar. The Python app connects through the sidecar's SOCKS5 proxy on 127.0.0.1:9050, which routes all HTTP traffic through the Tor network to reach .onion dark web forums and marketplaces.

Fargate is used instead of Lambda because the crawling engine maintains a persistent, stateful connection to the Tor control port for circuit rotation — Lambda's 15-minute limit and stateless cold-start model are incompatible with this.

Raw HTML content is stripped to plain text, hashed with SHA-256 for deduplication, and uploaded to Amazon S3 alongside a metadata sidecar. A circuit breaker pattern backed by DynamoDB tracks per-source failure counts — if a source fails 5 consecutive times, it's temporarily blocked to prevent hammering unreachable servers. Credentials for the Tor control port are retrieved at runtime from AWS Secrets Manager.

3

Intelligence Analysis — Lambda + Amazon Bedrock (Claude Opus 4) + Guardrails

Step Functions passes the S3 artifact key to the Content Analyst Lambda. In a single Amazon Bedrock API call — using Claude Opus 4 — the function simultaneously determines: (1) whether the content is fraud-relevant, (2) what entities are present (BINs, SWIFT codes, Bitcoin wallets, IPs, email addresses), and (3) which fraud technique category applies (MFA bypass, phishing kit, account takeover, CNP fraud, synthetic identity). Combining three analyses into one call reduces Bedrock costs by ~3× compared to sequential calls.

Bedrock Guardrails screen every input for prompt injection and harmful content before Claude processes it. If Guardrails intervene, the content is flagged as non-relevant and quarantined — the pipeline continues without exposing the model to potentially adversarial dark web text.

4

Intelligence Structuring — Lambda + STIX 2.1 + Bedrock Titan Embeddings + OpenSearch Serverless

The Data Structurer Lambda converts raw extracted entities into a STIX 2.1 Bundle — the international standard for threat intelligence. Entities are mapped to the correct STIX types: IP addresses become IPv4Address SCOs, institutions become Identity SDOs (victim organisations, not threat actors), attack patterns become AttackPattern SDOs, and all objects are linked via typed Relationship SROs.

Each STIX object is summarised into text, embedded using Bedrock Titan Embed, and indexed into an OpenSearch Serverless VECTORSEARCH collection. This enables future semantic similarity queries — "find all threat intelligence similar to this Bitcoin wallet pattern" — without full-text search limitations.

The serialised STIX bundle JSON is also written back to S3 for the downstream agents and for the Bedrock AgentCore Managed Knowledge Base, which enables RAG retrieval over historical threat intelligence in future agentic workflows.

5

Automated Tagging — Lambda + MITRE ATT&CK + MISP

The Tagging Engine Lambda applies machine-readable classification tags from three taxonomies: MITRE ATT&CK technique IDs (e.g. T1111 for MFA interception, T1566.001 for spearphishing attachments), a custom banking fraud taxonomy (fraud:type="cnp-fraud"), and severity level tags (low / medium / high / critical).

Items that don't match any taxonomy predicate are automatically tagged review:status="requires-review" for analyst triage. A tag manifest JSON is written to S3 alongside the STIX bundle. The Tagging Engine also pushes the STIX bundle to an external MISP instance — the open-source threat intelligence sharing platform — enriching it with MISP Galaxy cluster links to known threat actor profiles.

6

Alert Generation & Campaign Convergence — Lambda + DynamoDB + SNS → SQS → SIEM

The Alert Generator Lambda runs a campaign convergence check: if 3 or more intelligence items reference the same TTP within a 24-hour window, they are combined into a consolidated campaign alert. This threshold logic is backed by DynamoDB — each item is written with a TTL equal to the convergence window, so old items expire automatically without manual cleanup.

The Lambda is invoked both from Step Functions (scheduled path) and reactively from a DynamoDB Streams event source — meaning convergence is evaluated the moment a new item lands, not just on the next 5-minute cycle. High-severity items (score ≥ 7/10) trigger an immediate alert regardless of convergence count.

Alerts are published to an Amazon SNS topic and delivered to a FIFO SQS queue for ordered delivery to downstream SIEM/SOAR systems. FIFO guarantees that multiple alerts about the same TTP are processed in the correct sequence — critical for accurate threat timelines. A Dead Letter Queue (DLQ) captures any failed deliveries for forensic review.

🔐 Security Architecture

KMS Customer Managed Key
Every data store — S3, DynamoDB, Secrets Manager — is encrypted with a single CMK. Every decrypt call is logged to CloudTrail, creating a full audit trail of who accessed what intelligence and when.
Per-Agent IAM Roles
Each Lambda function has its own IAM role with least-privilege permissions. The Content Analyst can call Bedrock and read S3, but cannot write to DynamoDB. The Alert Generator can publish to SNS but cannot call Bedrock.
VPC Isolation
The Fargate crawling task runs in a private subnet. All Tor traffic egresses via a NAT Gateway to an Elastic IP — dark web traffic never flows from the Lambda analysis layer, which is completely separate.
GuardDuty + Security Hub
GuardDuty monitors for anomalous API calls and data exfiltration patterns. Security Hub aggregates findings from GuardDuty and Inspector (container vulnerability scans on the Fargate image) into a single SOC view.

📊 Observability

Every Lambda handler emits CloudWatch Embedded Metric Format (EMF) logs — JSON written to stdout that the Lambda runtime automatically converts to CloudWatch custom metrics. This gives the SOC dashboard real-time metrics on items processed per cycle, Bedrock token consumption, error rates per agent, and alert convergence events — with zero additional API calls. AWS X-Ray traces every Step Functions execution end-to-end, pinpointing latency across the Bedrock, S3, DynamoDB, and OpenSearch hops. Pipeline failure alarms (>3 failures / 5-min window) and DLQ depth alarms (>10 messages) route to the SNS ops topic, which can be subscribed by the SOC team's email or PagerDuty integration.
